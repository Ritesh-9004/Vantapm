Every feature, categorized by priority

01 Registry Website — Package Discovery
MUST — Day 1
Search engine
Full-text search by name, description, tag. Filter by: framework (Arduino/IDF/MicroPython), ESP32 variant (S3, C3...), category (sensor, display, protocol...), license type.
MUST — Day 1
Package detail page
Name, description, version, install command, README tab, Changelog tab, Example tab, dependency list, score breakdown, GitHub link, license, publisher.
MUST — Day 1
Quality score (0–100)
Auto-computed from: last commit age, open issues ratio, CI status, README completeness, has examples, has tests, download count, declared hardware targets. Shown prominently.
MUST — Day 1
ESP32 variant compatibility
Table showing: ESP32, S2, S3, C3, C6, H2, P4 — tested ✓ / untested ? / broken ✗. Auto-populated from CI or community reports.
MUST — Day 1
Memory footprint display
Shows Flash usage (KB) and RAM usage (KB) for each ESP32 variant. Color-coded warning if library would exceed 80% of target board's memory.
MUST — Day 1
Framework tags
Every package tagged: Arduino-compatible, ESP-IDF compatible, MicroPython compatible, or any combination. Filters on search page.
HIGH — v1.1
Verified publisher badge
Domain verification (like pub.dev). espressif.com packages get official badge. Companies can verify their domain to show trusted packages.
HIGH — v1.1
Dependency graph
Visual graph showing what this library depends on and what depends on it. Click any node to navigate to that package.
UNIQUE — v1.1
Arduino ↔ IDF bridge
"IDF equivalent" section on every Arduino library page — and vice versa. Shows recommended alternatives when switching frameworks.
UNIQUE — v1.2
Memory calculator
Interactive: select your board + your current libs → shows total Flash + RAM usage with a visual bar. Add library → see impact instantly before installing.
HIGH — v1.2
Version diff viewer
Compare any two versions side by side — what changed in the API, what breaking changes exist, memory delta between versions.
v2.0
Private registry
Teams can host private packages (company-internal components). Same API as public registry. SaaS pricing per seat.
02 CLI Tool (esp32pm)
MUST — Day 1
install / add
esp32pm install bme280 — fetches, downloads, pins version in esp32pm.toml, generates lockfile.
MUST — Day 1
remove / uninstall
Removes package and updates toml + lockfile. Warns if other packages depend on it.
MUST — Day 1
update
Updates all or specific packages respecting semver constraints. Shows changelog before updating.
MUST — Day 1
list
Lists installed packages with current version, latest available, and quality score.
MUST — Day 1
search
esp32pm search "temperature sensor" — queries registry, shows top 10 results with scores inline in terminal.
MUST — Day 1
init
Creates new esp32pm.toml for a project. Asks: board, framework, project name. Generates correct starting config.
UNIQUE — Day 1
check memory
esp32pm check memory — before installing, shows predicted Flash + RAM impact. Warns if exceeding budget defined in toml.
HIGH — v1.1
publish
Publish your library to the registry. Validates esp32pm.toml, runs lint, checks license, uploads tarball.
HIGH — v1.1
build
Wraps CMake + toolchain invocation. esp32pm build compiles for target board automatically.
HIGH — v1.1
translate
esp32pm translate --from arduino --to idf — shows IDF equivalents for all your Arduino dependencies.
03 Package Metadata Standard (esp32pm.json)
MUST
name, version, description
Standard package identity fields. Semver required. Description max 200 chars.
MUST
frameworks[ ]
Declares: "arduino", "espidf", "micropython", or combinations. The most critical metadata field for ESP32.
MUST
targets[ ] — ESP32 variants
Declares which ESP32 chips this works on: esp32, esp32s2, esp32s3, esp32c3, esp32c6, esp32h2, esp32p4.
MUST
dependencies{ }
List of other esp32pm packages with semver constraints. The foundation of dependency resolution.
HIGH
memory{ flash_kb, ram_kb }
Declared or auto-measured memory footprint per target. Used by memory calculator and install warnings.
HIGH
idf_version constraint
Minimum IDF version required. Critical since IDF v4.x → v5.x had breaking API changes that broke many libraries.
UNIQUE
peripherals[ ]
Which hardware peripherals this library uses: SPI, I2C, UART, ADC, DAC, LEDC, etc. Used for pin conflict detection.
UNIQUE
sensors[ ] / displays[ ]
What physical hardware components this library supports. Enables "find all libraries for BME280" search.

Step 04 — The manifest format
esp32pm.toml — your project's config file
Inspired by Cargo.toml (Rust) and pubspec.yaml (Flutter). TOML is chosen because it's simpler than YAML, more readable than JSON, and already familiar to Rust/embedded developers.
# esp32pm.toml — generated by: esp32pm init # ── Project Identity ────────────────────────────── [project] name = "weather_station" version = "1.0.0" authors = ["Your Name <you@email.com>"] # ── Target Hardware ─────────────────────────────── [target] board = "esp32-devkitc-v4" # from board registry chip = "esp32s3" framework = "arduino" # arduino | espidf | micropython toolchain = "xtensa-esp32s3-gcc@12.2.0" # LOCKED version # ── Memory Budget ───────────────────────────────── [budget] flash = "80%" # warn if total Flash exceeds 80% ram = "70%" # warn if total RAM exceeds 70% # ── Dependencies ────────────────────────────────── [dependencies] bme280_idf = "^2.4.0" FastLED_ESP32 = "^3.7.0" esp32_mqtt = "^1.2.0" ArduinoJSON = "^7.0.0" # ── Dev Dependencies (not included in firmware) ─── [dev-dependencies] unity_test = "^2.5.0" # unit testing framework # ── Build Options ───────────────────────────────── [build] optimization = "size" # size | speed | debug partition = "default" # default | no_ota | huge_app
Package metadata file (published with your library)
# esp32pm.json — required in every published package { "name": "bme280_idf", "version": "2.4.1", "description": "BME280 sensor driver for ESP32 (Arduino + IDF)", "license": "MIT", "repository": "https://github.com/sensor-labs/bme280_idf", // ← THE ESP32-SPECIFIC FIELDS pub.dev doesn't have "frameworks": ["arduino", "espidf"], "targets": ["esp32","esp32s2","esp32s3","esp32c3","esp32c6"], "idf_version": ">=4.4.0", "memory": { "flash_kb": 14.2, "ram_kb": 1.8 }, "peripherals": ["i2c", "spi"], "sensors": ["BME280", "BMP280"], "category": "sensor", "dependencies": {} }
Step 05 — What makes us different from pub.dev
The ESP32-specific features pub.dev never needed
pub.dev works for one platform. We work for hardware. These features don't exist in any package manager today.

Quality / scoring system (likes + points)
Verified publisher domain badges
README / Changelog / Example tabs

Lock file (esp32pm.lock)
Dependency graph visualization
CLI install / update / publish
Issue count + PR activity signal
Pre-release version support
WHAT WE ADD → ESP32-EXCLUSIVE
Hardware compatibility matrix (per ESP32 variant)
Flash + RAM footprint per chip
Memory budget warning before install
