The complete ESP32 package landscape — 6 separate worlds

500+

ESP-IDF Component Registry

Has central PM

components.espressif.com — Official Espressif registry. Proper CLI, semver, dep resolution, lock files. Best designed. But ESP-IDF only — Arduino devs can't use it at all.

1,219

Arduino Library Manager

Partial PM (GUI only)

arduinolibraries.info — 1,219 ESP32 tagged libs. GUI-only inside Arduino IDE. No CLI, no dep resolution, no lock files, no memory awareness. Separate from IDF registry entirely.

~2,000

PlatformIO Registry

Partial PM

registry.platformio.org — Has CLI and version pinning. Covers both Arduino + IDF. But poor quality control, many abandoned libs. Doesn't bridge to Espressif's own component registry.

10,000+

GitHub — Raw Libraries

NO central PM

The REAL majority of ESP32 libraries. No metadata, no versioning, no quality signal. Developers manually clone repos and copy folders. This is the biggest gap.

300+

MicroPython (mip)

Has mip (basic)

micropython-lib on GitHub. Installs over WiFi onto chip. Tiny registry, no dep resolution, no version locking. Completely separate from C/C++ ecosystem.

400+

ESPHome Registry

YAML only — walled garden

esphome.io/components — 400+ YAML components. If you write C/C++ firmware, this ecosystem is completely inaccessible to you.

Every category where NO PM exists for ESP32

CriticalSensor & peripheral drivers — 3,000+ GitHub repos, zero central PM▼
Every ESP32 project uses sensors. There are thousands of drivers on GitHub — each by a different person, different quality. No central registry. Developers Google "ESP32 BME280 library" and manually pick one hoping it's maintained. Sensors with NO central PM: Temp/Humidity: DHT11, DHT22, SHT30, BME280, AHT20... Pressure: BMP280, BME680, MS5611... Distance: HC-SR04, VL53L0X, VL53L5CX, TFMini... Motion/IMU: MPU6050, ICM42688, BNO055, ADXL345... Display: ILI9341, ST7789, SSD1306, GC9A01, EPD... Each has 5–20 different GitHub repos. No way to find the best one.
CriticalArduino ↔ ESP-IDF bridge — two ecosystems that never talk to each other▼
ESP-IDF registry and Arduino library manager are completely separate. Switching from Arduino to IDF means manually finding equivalents for every library. Switching cost keeps developers stuck on Arduino. Today: "switch to IDF" = 1–3 days searching for equivalents per project With bridge PM: mcupm translate --from arduino --to espidf ArduinoJSON TFT_eSPI → Done in 30 seconds with verified equivalents
CriticalAI / ML on-chip libraries — fastest growing area, zero PM coverage▼
ESP32-S3 and P4 have AI acceleration. TFLite Micro, ESP-DL, ESP-WHO, Edge Impulse SDK, pre-quantized models — all manually installed. Entire TinyML pipeline is manual downloads and CMake fighting. AI/ML ESP32 libs with ZERO PM: TFLite Micro: manual CMake ESP-DL, ESP-WHO: no IDF component, manual git clone Edge Impulse SDK: manual zip download per project Pre-quantized models: shared via Google Drive or GitHub releases
MajorAudio & DSP libraries — 100% GitHub-only, zero PM coverage▼
ESP32 is widely used for audio. ESP-ADF exists but is installed via git clone. Codecs (MP3, AAC, FLAC), DSP effects (reverb, EQ, compressor), voice (wake word, VAD) — all GitHub-only. Audio/DSP on GitHub with no PM: minimp3, helix-mp3, libopus-esp32, FLAC decoder reverb/EQ/compressor effect libs ESP-SR (speech recognition) — official but no IDF component PM
MajorIndustrial protocols — companies suffer most, 100% scattered▼
Modbus RTU/TCP, BACnet, OPC-UA, CANopen, PROFIBUS gateway, KNX — all critical for industrial ESP32 use. Essentially 100% GitHub-only with no versioning, no CVE tracking, no quality signal. Industrial protocols with NO PM: Modbus RTU/TCP: multiple GitHub repos, none authoritative BACnet: 1-2 experimental repos, unmaintained OPC-UA: open62541 port exists on GitHub, manual build CANopen: GitHub only, quality varies wildly
MajorMotor & actuator control — robotics developers reinvent every project▼
SimpleFOC, stepper drivers (TMC2209, A4988), BLDC control, encoder reading, CNC (Grbl-ESP32) — all scattered. No PM means no reproducibility and random breakage after upstream git changes.
GrowingSecurity & crypto — critical for production, completely invisible▼
Secure provisioning wrappers, certificate management, eFuse key management, secure OTA signing — all GitHub-scattered. Companies shipping production ESP32 devices have no standard way to manage these critical security components.

Every type of ESP32 developer and what they need

Maker / Hobbyist
The Weekend Builder
Uses Arduino. Needs sensor libs, display libs, WiFi helpers. Can't tell which of 8 DHT libs is maintained. Picks the wrong one by accident every time.
Pain: No quality scores in Arduino Library Manager
Student
The CS/ECE Student
Wants one command to install everything. Gets confused by two separate registries (Arduino vs IDF). Doesn't understand why the same sensor has 6 different libs.
Pain: Two completely separate registries with no beginner guidance
Professional Dev
The Firmware Engineer
Uses ESP-IDF. Needs reproducible team builds, version locking, CI/CD, private component registry for internal company libs.
Pain: No private registry hosting for internal company components
IoT Builder
The Startup Founder
Shipping a product. Needs OTA, MQTT, security, provisioning. These exist as separate GitHub repos with no guarantee they work together.
Pain: 6 GitHub repos with no integration guarantee
Industrial Dev
Factory Automation Engineer
Uses ESP32 with Modbus, CAN bus. Manually maintains GitHub repo copies in SVN. Has to audit security vulnerabilities by hand for each library.
Pain: Industrial protocol libs have no versioning or CVE tracking
Robotics
The Robot Maker
SimpleFOC, encoders, servos — all GitHub. No lockfile means upstream git changes randomly break projects after git pull.
Pain: No lockfile for GitHub-sourced libs
AI Developer
TinyML Engineer
TFLite Micro, ESP-DL, pre-quantized models — all manual installs. Every new model is a 2-hour CMake setup process. Nothing is packageable yet.
Pain: Entire AI pipeline is 100% manual — no PM exists
Audio Dev
Audio Tinkerer
ESP-ADF has no PM. Additional codecs and DSP filters are GitHub repos. No way to declare dependencies across ESP-ADF + codecs + effects.
Pain: ESP-ADF = manual git clone + CMake editing every project
The single most valuable thing to build first: A unified ESP32 driver registry with quality scores — targeting the 3,000+ scattered sensor/peripheral libs.

Every ESP32 developer, every day, wastes 20–40 minutes finding a sensor driver and picking between 8 GitHub repos. A registry that indexes ALL ESP32 sensor libs from GitHub automatically, gives quality scores, tags Arduino vs IDF compatibility, and lets you install with one command — esp32pm install bme280 --framework arduino — would be used by millions immediately. This exists nowhere today.
