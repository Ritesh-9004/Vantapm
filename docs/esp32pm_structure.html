For each package — exactly 4 things download

esp32pm.json
METADATA — ~2KB
Package manifest. Contains: name, version, declared targets, framework tags, memory footprint, dependencies list, peripheral declarations. Downloaded first, before source code. Used to resolve dependency tree without downloading everything.
downloads instantly — tiny JSON file
source.tar.gz
SOURCE CODE — 10KB–5MB
The actual library source code. Contains: all .c / .cpp / .h files, the library's own CMakeLists.txt, examples/, LICENSE, README.md. Extracted into .esp32pm/packages/name@version/ automatically.
extracted after download — never left as .tar.gz
checksum.sha256
INTEGRITY — 64 bytes
SHA256 hash of the source tarball. Downloaded alongside every package. Verified before extraction — if hash doesn't match, install aborts with a clear error. Prevents tampered or corrupted packages.
verified before extraction — security critical
toolchain tarball
COMPILER — 200MB–1GB
Downloaded ONCE per toolchain version and cached globally. xtensa-esp32s3-gcc@12.2.0 = ~400MB. ARM GCC for bare-metal = ~280MB. If another project on your machine uses the same toolchain version, it reuses the cache — no re-download.
heaviest download — cached globally, never per-project

What does NOT download

Examples & Tests (optional)
OPTIONAL — skipped by default
Library examples and test files are NOT downloaded by default. Only downloaded if you explicitly run esp32pm install --with-examples. Saves disk space when you just need the library to work.
Docs / Wiki
NEVER DOWNLOADED
HTML documentation, GitHub wikis, Doxygen output — never downloaded. The website serves docs online. No reason to put megabytes of HTML on your disk. esp32pm docs bme280 opens the browser instead.
Dev dependencies (of libraries)
NEVER DOWNLOADED
If bme280_idf uses unity_test for its own testing — you don't need that. Dev dependencies of your dependencies are never installed. Only runtime dependencies are resolved and downloaded.
Every File — Explained
What each file is, who creates it, what's inside it, and the rules around it

Files YOU create and own

esp32pm.toml
YOU WRITE — ALWAYS COMMIT
The heart of your project. Declares: board, chip, framework, toolchain version, dependencies with semver ranges, memory budget limits. This is the ONLY config file you manually write. Everything else is generated from it.
✓ commit ✓ edit freely ✗ never auto-edited by esp32pm
esp32pm.lock
AUTO-GENERATED — ALWAYS COMMIT
Generated by esp32pm after resolving your dependency tree. Contains EXACT versions of every package including transitive deps. Your teammate runs esp32pm install and gets bit-identical packages. Never edit by hand.
✓ commit ✗ never edit manually ✓ regenerated by esp32pm update
src/main.cpp
YOU WRITE
Your firmware code. You #include library headers normally — no path hacks. esp32pm configures CMake so that #include "bme280.h" just works. Your code stays clean — no ../../../libraries/BME280/src nonsense.
✓ commit ✓ edit freely
include/config.h
YOU WRITE
Your project-wide configuration. Pin definitions, WiFi credentials (use env vars!), feature flags. These are YOUR headers — not library headers. Kept separate from src/ so any .cpp can include them easily.
✓ commit (except secrets — use .env)

Files esp32pm auto-generates (never touch)

.esp32pm/cmake/esp32pm_deps.cmake
AUTO-GENERATED
The CMake glue file. Contains add_subdirectory() calls for every installed package, sets include paths, links libraries. Regenerated every esp32pm install. Your CMakeLists.txt just includes this one file.
✗ git-ignored ✗ never edit ✓ auto-regenerated
.esp32pm/cmake/toolchain_xtensa.cmake
AUTO-GENERATED
CMake toolchain file. Tells CMake which compiler to use, target architecture, CPU flags (-mcpu=esp32s3, -mfpu=...), sysroot path. Generated from your target.chip + target.toolchain in esp32pm.toml.
✗ git-ignored ✗ never edit ✓ auto-regenerated
.esp32pm/cmake/CMakeLists.txt
AUTO-GENERATED
The root CMakeLists.txt for your project. Sets project name, C/C++ standards, includes the toolchain file, adds your src/ files, includes esp32pm_deps.cmake. If you need custom CMake — add a CMakeExtras.cmake file in your root and esp32pm will include it.
✗ git-ignored ✗ never edit ✓ auto-regenerated
.esp32pm/install_state.json
AUTO-GENERATED
Internal state file. Tracks which packages are installed, their exact paths, SHA256 hashes, and timestamps. Used by esp32pm to detect if packages were corrupted or manually modified. Do not touch.
✗ git-ignored ✗ never edit
.esp32pm/memory_snapshot.json
AUTO-GENERATED
Stores measured Flash + RAM contribution of each installed package. Used by esp32pm check memory and the VS Code memory bar. Updated after every build by parsing the .map file output.
✗ git-ignored ✗ never edit
build/firmware.bin
COMPILED OUTPUT — GIT-IGNORED
The actual flashable firmware binary. Produced by CMake + GCC after successful build. Flash to ESP32 with esp32pm flash (wraps esptool). Never commit — always reproducible from source.
✗ git-ignored ✗ never commit

Why a global cache matters

ScenarioWithout global cacheWith global cache
Project 1 installs ArduinoJSON@7.0.4Download 180KB tarballDownload 180KB tarball → cached
Project 2 also needs ArduinoJSON@7.0.4Download againInstant — already in cache
Clone project on same machine, run installDownload all packages againInstant — all in cache
Toolchain download (400MB)400MB per project (!)400MB once, symlinked forever
CI/CD pipeline cold startDownload everything (slow)Restore cache dir → instant
Offline developmentFails — no internetWorks — reads from cache
Cache size management: esp32pm cache clean --older-than 90d removes packages not used in 90 days. esp32pm cache info shows total disk usage. Typical cache after 6 months of work: 2–8GB (mostly toolchains).

Why commit esp32pm.lock?

Yes — always commit the lock file. The lock file records the EXACT resolved version of every package (including transitive deps). Without it, running esp32pm install one week later might give you a slightly different version of a transitive dependency — and your firmware breaks in a mysterious way. Committing the lock file means every developer and every CI run gets exactly the same dependency tree, forever. This is how npm, Cargo, and pub.dev all work.

Why NOT commit .esp32pm/?

.esp32pm/ is fully reproducible from esp32pm.toml + esp32pm.lock. It's like committing your node_modules/ — wasteful, causes merge conflicts, and breaks cross-platform compatibility (a .so compiled on Linux won't work on macOS). Run esp32pm install and it's regenerated in seconds from the global cache.
Special case — vendoring for offline/safety: If your company requires fully offline builds or has strict supply chain requirements, use esp32pm vendor which copies all packages into a vendor/ folder that IS committed to git. This is the embedded industry's equivalent of "vendoring" in Go.
