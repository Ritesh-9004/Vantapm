Database Decision
Supabase vs Railway vs Neon — what to actually use for omnilink

The 3 real options evaluated

Supabase
SKIP FOR NOW
Supabase = PostgreSQL + Auth + Storage + Realtime + Edge Functions + Auto-generated API all bundled together.

For omnilink this is OVERKILL. You are building your own API with Hono. You don't need Supabase's auto-generated REST API — it conflicts with your custom API. You don't need their auth layer — you'll build your own token system. You're paying for 5 features you won't use.

Free tier pauses after 7 days of inactivity — breaks the crawler. Pro is $25/month. Too much for a DB you're only using as plain Postgres.
USE THIS
Railway
RECOMMENDED
Railway = just PostgreSQL. Pay for what you use. No bundled features you don't need. Runs your API + DB + Redis + Python crawler all in one place from one dashboard.

$5/month Hobby covers everything for v1. Never pauses. One-click Postgres deploy. Automatic backups on Pro. Direct DATABASE_URL connection string. Works perfectly with Drizzle ORM.

You already chose Railway for the full stack. Keep everything in one place.
Neon
GOOD ALTERNATIVE
Neon = serverless PostgreSQL. Scales to zero when idle (free). Instant database branching — clone your prod DB for testing in seconds. Great for solo developers.

The "scales to zero" feature means a cold start delay (0.5–3s) on first query. For a package registry that serves millions of package downloads, cold starts are bad. Railway's always-on Postgres is better for a registry workload.

Consider Neon for development branches, Railway for production.

The rule: Use Supabase when you want their whole backend bundle. Use Railway Postgres when you just need a great PostgreSQL database and nothing else. For omnilink — you need PostgreSQL, your own API, Cloudflare R2. Railway is the right fit.

001_packages.sql
-- Core package registry table -- One row per UNIQUE package name (not per version) CREATE TABLE packages ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), name TEXT NOT NULL UNIQUE, description TEXT, repository_url TEXT, -- github.com/user/repo homepage_url TEXT, license TEXT, -- MIT, Apache-2.0, GPL-3.0 category TEXT, -- sensor, display, protocol, rtos, etc -- Framework support flags supports_arduino BOOLEAN DEFAULT FALSE, supports_espidf BOOLEAN DEFAULT FALSE, supports_micropython BOOLEAN DEFAULT FALSE, supports_stm32hal BOOLEAN DEFAULT FALSE, -- phase 2 supports_zephyr BOOLEAN DEFAULT FALSE, -- future -- GitHub signals (updated by crawler) github_stars INTEGER DEFAULT 0, github_forks INTEGER DEFAULT 0, open_issues INTEGER DEFAULT 0, closed_issues INTEGER DEFAULT 0, has_ci BOOLEAN DEFAULT FALSE, -- has .github/workflows/ has_readme BOOLEAN DEFAULT FALSE, has_examples BOOLEAN DEFAULT FALSE, -- has examples/ folder has_tests BOOLEAN DEFAULT FALSE, last_commit_at TIMESTAMPTZ, -- Denormalized latest version (for fast listing) latest_version TEXT, total_downloads BIGINT DEFAULT 0, -- Publisher publisher_id UUID REFERENCES publishers(id), is_official BOOLEAN DEFAULT FALSE, -- published by espressif, st, etc is_verified BOOLEAN DEFAULT FALSE, -- domain-verified publisher -- Full-text search vector (auto-updated) search_vector TSVECTOR, -- Source tracking source TEXT DEFAULT 'github', -- github | manual | official crawled_at TIMESTAMPTZ, created_at TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW() ); -- Full-text search index (name + description weighted) CREATE INDEX idx_packages_search ON packages USING GIN(search_vector); -- Auto-update search vector on insert/update CREATE OR REPLACE FUNCTION update_search_vector() RETURNS TRIGGER AS $$ BEGIN NEW.search_vector := setweight(to_tsvector('english', coalesce(NEW.name, '')), 'A') || setweight(to_tsvector('english', coalesce(NEW.description, '')), 'B') || setweight(to_tsvector('english', coalesce(NEW.category, '')), 'C'); RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER packages_search_update BEFORE INSERT OR UPDATE ON packages FOR EACH ROW EXECUTE FUNCTION update_search_vector(); -- Common query indexes CREATE INDEX idx_packages_category ON packages(category); CREATE INDEX idx_packages_arduino ON packages(supports_arduino) WHERE supports_arduino = TRUE; CREATE INDEX idx_packages_espidf ON packages(supports_espidf) WHERE supports_espidf = TRUE; CREATE INDEX idx_packages_downloads ON packages(total_downloads DESC);

Step-by-step local setup

01
Clone and install dependencies
Get the monorepo running. pnpm handles all workspaces.
git clone https://github.com/[you]/omnilink
cd omnilink
pnpm install # installs all workspaces
02
Start Postgres + Redis locally
Docker Compose spins up the database. First run auto-executes migrations in initdb.d/
docker compose up -d # starts postgres + redis in background
docker compose ps # verify both are healthy
03
Set up environment variables
Copy the example env file. Never commit .env
cp apps/api/.env.example apps/api/.env
# .env contents:
DATABASE_URL=postgresql://omnilink:omnilink_dev@localhost:5432/omnilink_db
REDIS_URL=redis://localhost:6379
GITHUB_TOKEN=ghp_your_token_here # for crawler rate limits
R2_BUCKET=omnilink-packages # Cloudflare R2
R2_ACCESS_KEY=your_key
R2_SECRET_KEY=your_secret
04
Run database migrations
Drizzle applies all SQL migration files in order
cd apps/api
bun run db:migrate # runs: drizzle-kit push
# OR for explicit migration files:
bun run db:generate # generates SQL from schema.ts
bun run db:push # applies to local postgres
05
Seed with test data
Insert 50 real ESP32 packages to develop against — don't run the full crawler locally
bun run db:seed # inserts sample packages, boards, scores
# seeds: 50 packages, 200 versions, board definitions
06
Start the development servers
Turborepo starts all apps simultaneously with hot reload
pnpm dev # from repo root — starts everything
# API: http://localhost:3001
# Website: http://localhost:3000
# DB: postgresql://localhost:5432/omnilink_db
