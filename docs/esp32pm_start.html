Where to Start
You've done the thinking. Now here's the exact first move — no wasted effort.

The single most important decision first

Start with the Registry Website + Auto-indexer. NOT the CLI.

Here's why: the website gives you something to SHOW people immediately. You can share a URL, get feedback, build an audience, and prove the concept — all before writing a single line of CLI code. The CLI without a registry is useless. The registry without a CLI is still useful (people can browse and discover libraries). Build the thing that proves value first.

Your first 4 weeks — exact tasks in order

WeekTaskOutputWhy this order
Week 1 Database schema + GitHub crawler PostgreSQL DB with 500+ ESP32 libs auto-imported from GitHub Data first — without packages, the website has nothing to show
Week 2 Registry API (backend) REST API: GET /packages, GET /packages/:name, GET /search API before UI — decouples frontend from backend work
Week 3 Registry website (frontend) Working website: search, package list, package detail page Now you have something people can actually use and share
Week 4 Quality scoring system Every package gets a 0–100 score based on GitHub signals This is your first killer feature — immediate differentiation
Do NOT start with the CLI. The CLI needs the registry to be useful. The website needs the database. The database needs the crawler. Start at the bottom of the stack and work up.

Tech Stack — Every Decision Explained
Chosen for speed of building, not theoretical perfection. You can ship faster with this stack than anything else.

CLI Tool

MUST USE
CLI LANGUAGE
Rust
cargo, clap, tokio, reqwest, serde
Fast cold start (critical for CLI — nobody wants to wait 2s for a command). Single binary, no runtime to install. Cross-compiles to Windows/macOS/Linux from one codebase. The embedded community already respects Rust deeply — using it signals credibility. Cargo itself is proof Rust makes world-class package managers.
Alt considered: Go (fast but weaker ecosystem), Python (slow startup, needs runtime)
MUST USE
DEPENDENCY RESOLVER
PubGrub Algorithm
pubgrub crate (Rust)
The gold standard dependency resolution algorithm — same one used by pip and Cargo. Guarantees finding the optimal solution OR gives a clear human-readable explanation of why no solution exists. DO NOT write your own resolver. This crate exists, it's tested, use it.
Never write your own — it's a PhD-level CS problem
RECOMMENDED
BUILD SYSTEM
CMake + Ninja
cmake 3.25+, ninja-build
CMake is the only cross-platform build system with proper ARM/Xtensa toolchain file support. Your CLI generates CMakeLists.txt from esp32pm.toml — users never write CMake. Ninja makes it fast. This is what ESP-IDF and Zephyr both use internally.
Not Make (too old), not Bazel (too complex for embedded beginners)

Registry Backend (API)

RECOMMENDED
API FRAMEWORK
Hono TypeScript
hono, bun runtime, zod validation
Extremely fast, lightweight, TypeScript-native. Runs on Bun (fastest JS runtime). Edge-deployable later if needed. Type-safe from day one. Much simpler than NestJS for an API of this size. You'll write 3x less boilerplate than Express.
Alt: Fastify (good), Express (too old), NestJS (overkill for v1)
MUST USE
DATABASE
PostgreSQL 16 SQL
drizzle-orm, postgres.js
Full-text search built in (pg_tsvector) — no need for Elasticsearch at first. JSONB columns for flexible package metadata. Reliable, battle-tested, free. Drizzle ORM gives you type-safe queries with zero magic. Same stack npm and crates.io use.
NOT MongoDB — structured package data belongs in SQL
RECOMMENDED
FILE STORAGE
Cloudflare R2
S3-compatible API, free egress
Store package tarballs (.tar.gz files). S3-compatible so your code is portable. Cloudflare R2 has ZERO egress fees — critical since packages get downloaded millions of times. Cheaper than AWS S3 for a package registry with high download volume.
Alt: AWS S3 (egress costs will hurt), Backblaze B2 (also good)
RECOMMENDED
CACHE LAYER
Redis (Upstash)
ioredis, rate limiting, session
Cache hot package metadata — /packages/bme280 gets called thousands of times per day. Rate limit the API per IP. Store publish tokens temporarily. Upstash is serverless Redis — no server to manage, pay per request.
Can skip in v1 — add when you hit performance issues

Registry Website (Frontend)

MUST USE
WEB FRAMEWORK
Next.js 15 React
App Router, Server Components, TypeScript
Server-side rendering for SEO — package pages need to be crawlable by Google. Static generation for docs pages. App Router gives you streaming and React Server Components. Vercel deploys it for free. Most importantly — you probably already know React.
Alt: SvelteKit (great but smaller ecosystem), Remix (also good)
RECOMMENDED
STYLING
Tailwind CSS + shadcn/ui
tailwindcss, shadcn/ui, lucide-react
Tailwind for custom styling speed. shadcn/ui gives you production-quality components (search, tables, dialogs, tabs) that you own and can customize — not a locked-in component library. You saw the dark terminal aesthetic in the mockup — Tailwind + custom CSS variables makes this easy.
Don't use MUI or Chakra — they'll make it look generic
RECOMMENDED
SEARCH UX
Algolia (free tier) or Meilisearch
instantsearch.js, react-instantsearch
Real-time search-as-you-type with typo tolerance. Algolia's free tier covers 10K searches/month — enough for v1. Meilisearch is self-hosted alternative (free, fast, open source). Both give you the pub.dev style "results appear instantly" search UX.
Can start with PostgreSQL full-text search (pg_tsvector) to avoid extra dependency

GitHub Crawler (Auto-Indexer)

RECOMMENDED
CRAWLER
Python 3.12 Python
PyGithub, httpx, asyncio, APScheduler
GitHub API is well-supported in Python. asyncio for parallel requests (respect rate limits while crawling fast). APScheduler runs the crawler every 6 hours automatically. Python is faster to write one-off scripts in than TypeScript. This is a background service — startup speed doesn't matter.
Alt: Node.js with octokit (also fine)
RECOMMENDED
QUALITY SCORER
Python script
PyGithub, semver, custom scoring logic
Reads GitHub API signals: last_commit_date, open_issues_count, has_ci (Actions), star_count, has_readme, has_examples, has_tests. Computes 0–100 score. Runs after every crawl cycle. Stores results in PostgreSQL. Can use basic LLM call to auto-categorize library type (sensor/display/protocol).
This is your most valuable first feature — do not skip it
FLEXIBLE
JOB QUEUE
BullMQ or simple cron
bullmq, redis, node-cron
For v1 — a simple cron job that runs the Python crawler script every 6 hours is fine. For v2 when you have thousands of packages updating constantly — BullMQ with Redis gives you a proper job queue with retries, priorities, and monitoring.
Start simple: just a cron job. Add queues when you need scale.

Infrastructure

RECOMMENDED
HOSTING
Vercel + Railway
vercel (web), railway (api + db + redis)
Vercel for Next.js (free tier, automatic deploys from GitHub). Railway for everything else — PostgreSQL, the Hono API, Redis, the Python crawler. Railway is $5/month and handles all your backend in one place. No DevOps needed at all in v1.
Alt later: AWS / GCP when you need enterprise features
RECOMMENDED
CI/CD
GitHub Actions
actions/checkout, bun setup, rust toolchain
Free for public repos. Auto-deploy on push to main. Run tests before deploy. Build and release the Rust CLI binary for all platforms automatically. Build Docker images for the crawler. This is how PlatformIO does it and it works perfectly.
Circular benefit: your users use GitHub Actions, so your tool must work there
RECOMMENDED
MONOREPO TOOL
Turborepo + pnpm
turbo, pnpm workspaces
All your TypeScript code (web, api, shared types) in one repo. pnpm workspaces link packages together. Turborepo caches builds — only rebuilds what changed. Share types between frontend and backend without publishing to npm. Rust CLI lives in the same repo as a separate workspace.
Could use Nx — Turborepo is simpler for this project size

Why a monorepo? The web app and the API share the same TypeScript types for Package, Version, QualityScore etc. With a monorepo + pnpm workspaces, you import from "@esp32pm/types" in both — no publishing to npm, no version mismatch, instant updates everywhere. The Rust CLI lives in the same repo but is a completely independent Cargo workspace.
Build Roadmap
Exact phases, tasks, and the reasoning behind every ordering decision.
01
Foundation — Database + Crawler + Basic Website
Weeks 1–4 — Build alone or with 1 other
▼
Why start here: Data before UI. A beautiful website with no packages is useless. A simple website with 1,000 real packages is immediately valuable. Get real data first, build the pretty parts second.
PostgreSQL schema
Tables: packages, versions, quality_scores, boards, frameworks. Full-text search indexes. Drizzle migrations.
apps/api/src/db/schema.ts
GitHub crawler v1
Search GitHub for repos tagged "esp32", "arduino-esp32", "espidf". Pull metadata: stars, last_commit, description, topics, has_ci, has_readme.

Metadata extractor
Parse library.properties (Arduino format), idf_component.yml (IDF format), and any esp32pm.json if present. Fall back to README parsing with regex.

Quality scorer v1
Score formula: last_commit (20pts) + has_ci (20pts) + readme_length (15pts) + star_count (15pts) + has_examples (15pts) + issues_ratio (15pts).
apps/crawler/quality_scorer.py
REST API — read only
GET /packages (list, paginated, sortable), GET /packages/:name (detail), GET /search?q= (full-text). No auth needed yet.

Search bar, package cards with quality score badge, framework tags, filter sidebar. Exact design from the mockup we built.

Name, description, install command, quality breakdown, GitHub stats, framework tags, version history. No compatibility matrix yet.
apps/web/app/packages/[name]/page.tsx
Deploy to production
Vercel for web, Railway for API + PostgreSQL. GitHub Actions: test on PR, deploy on merge to main. Custom domain: esp32pm.dev

02
Killer Features — Compat Matrix + Memory Widget + CLI v1
Weeks 5–10 — This is where you differentiate
▼
Why this is phase 2: Once the basic site is live and getting some users, these are the features that make people tell their friends. The compatibility matrix and memory widget don't exist anywhere else. Ship them fast while momentum is building.
ESP32 variant compat matrix
QEMU-based CI: build each library for ESP32, S2, S3, C3, C6. Record pass/fail. Show green/red matrix on package page. Start with QEMU simulation, add real boards later.
apps/crawler/compat_tester.py
Memory footprint measurement
Build each lib for each target, parse .map file output, extract text section size (Flash) and bss+data (RAM). Store in versions table. Show on package page.
apps/crawler/memory_measurer.py
Memory widget (frontend)
Interactive memory bar on package page. Select your board → see Flash + RAM impact. Add multiple packages → see cumulative total. This is your #1 viral feature.
apps/web/components/MemoryWidget.tsx
Compat matrix component
Color-coded table: ✓ tested / ? unknown / ✗ broken per ESP32 variant per framework. Click cell → see test log.
apps/web/components/CompatMatrix.tsx
CLI v1 (Rust) — install + search
esp32pm search [query], esp32pm install [package], esp32pm list. Reads/writes esp32pm.toml. Downloads to .esp32pm/packages/. No CMake generation yet.
apps/cli/src/
esp32pm.toml parser (Rust)
Parse the project config file. Validate board, framework, dep semver ranges. Serde + toml crate. This is the foundation all other CLI commands build on.
apps/cli/src/config/mod.rs
Lock file generation (Rust)
Run pubgrub resolver against the registry. Write esp32pm.lock with exact resolved versions. On reinstall, read lock file for exact versions.
apps/cli/src/resolver/mod.rs
Arduino ↔ IDF bridge data
Manually curate a JSON mapping: top 100 Arduino libs → their IDF equivalents. Show "IDF equivalent" section on Arduino package pages. Ship the data before building fancy auto-detection.
packages/core/src/framework_bridge.json
03
Complete CLI — Build + Flash + CMake generation
Weeks 11–16 — Making the full workflow work
▼
Why phase 3: The CLI becomes truly useful only when it handles the FULL workflow — not just install but also build and flash. CMake generation is the hardest engineering task in the whole project. Do it after the website proves the concept.
CMake generator (Rust)
Generate .esp32pm/cmake/esp32pm_deps.cmake, toolchain file, root CMakeLists.txt from esp32pm.toml. This is the hardest single task in the project.
apps/cli/src/cmake/generator.rs
Toolchain manager (Rust)
Download xtensa-esp32s3-gcc@12.2.0 tarball to global cache. Verify SHA256. Extract. Create symlink in project. Support: ESP32, S2, S3, C3, C6 toolchains.
apps/cli/src/toolchain/manager.rs
esp32pm build command
Calls cmake --configure then ninja. Shows progress. Parses .map output for memory report. Shows Flash/RAM breakdown after build.
apps/cli/src/commands/build.rs
esp32pm flash command
Wraps esptool.py. Auto-detects USB port. Flashes firmware.bin. Opens serial monitor. Works on Windows, macOS, Linux.
apps/cli/src/commands/flash.rs
Memory budget checker
Before install: check if adding this lib would exceed flash_budget or ram_budget in esp32pm.toml. Show warning with alternatives if it would. The killer CLI feature.
apps/cli/src/memory/checker.rs
Global cache management
~/.esp32pm/ with package cache, toolchain cache, registry metadata cache. esp32pm cache info (show disk usage), esp32pm cache clean --older-than 90d.
apps/cli/src/cache/mod.rs
Publish flow (API + CLI)
esp32pm publish: validates esp32pm.json, builds tarball, uploads to R2, creates DB record. API: POST /packages with auth token. Auth: email/token login.
apps/cli/src/commands/publish.rs
VS Code extension v1
Sidebar showing installed packages with quality scores and memory bars. One-click install from VS Code. Calls CLI binary under the hood.
apps/vscode-ext/src/
04
Growth — Community + Private Registry + Monetization
Month 5+ — After you have real users
▼
Why phase 4 last: Don't build monetization before you have users. Don't build a private registry before companies are asking for it. Phases 1–3 get you users. Phase 4 turns users into a business.
Private registry (SaaS)
Teams host private packages. Same API as public registry. Per-seat pricing. This is your main revenue source — companies pay for internal component management.
User accounts + publisher profiles
Sign up, claim your GitHub packages, verified publisher badge. Like npm's "verified" or pub.dev's publisher verification.
SBOM generation
esp32pm sbom — generates a Software Bill of Materials in SPDX format. Required by US EO 14028 for connected devices. Huge enterprise selling point.
Real hardware CI farm
Physical ESP32 boards in CI. Real hardware test results on the compat matrix. Start with 5 boards (ESP32, S3, C3, C6, P4), expand as revenue allows.
esp32pm translate command
esp32pm translate --from arduino --to espidf — shows IDF equivalents for all your deps. Auto-suggests migration path when switching frameworks.
Security scanning
CVE scanning per package. Alert package page if a vulnerability is found. esp32pm audit command. Critical for enterprise customers.

"Build the GitHub crawler for esp32pm. It should search GitHub for repos with topics: esp32, arduino-esp32, espidf, esp-idf, esp32-library. For each repo, collect: name, description, stars, last_commit_at, open_issues_count, has_ci (has .github/workflows/), has_readme, has_examples (examples/ folder exists), topics, license. Respect GitHub rate limit (5000 req/hr authenticated). Use asyncio for parallel fetching. Store to PostgreSQL via the esp32pm API POST /internal/packages endpoint. Run every 6 hours via APScheduler."
WHEN STARTING API
Registry API Prompt
"Build the esp32pm registry API using Hono on Bun. Endpoints needed: GET /packages (paginated list, sortable by quality/downloads/updated, filterable by framework/category/target), GET /packages/:name (full package detail with all versions), GET /packages/:name/:version (specific version), GET /search?q=&framework=&category= (full-text search via pg_tsvector). Use Drizzle ORM with PostgreSQL. Return snake_case JSON. Include quality_score and memory fields in every package response."
WHEN STARTING WEBSITE
Registry Website Prompt
"Build the esp32pm registry website in Next.js 15 with App Router. Dark terminal aesthetic — black background (#0d1117), green accent (#3fb950), IBM Plex Mono for code, Syne for headings. Pages: / (homepage with search + package list), /packages/[name] (package detail). Package cards show: name, description, quality score badge (0-100), framework tags (Arduino/IDF), memory footprint, last updated. Package detail shows: install command with copy button, ESP32 variant compat matrix table, memory widget bars, quality score breakdown."
WHEN STARTING CLI
Rust CLI Prompt
"Build the esp32pm CLI in Rust. Use clap for argument parsing, reqwest+tokio for async HTTP, serde+toml for config parsing. First implement: `esp32pm search [query]` (calls GET /search, pretty-prints results in terminal), `esp32pm init` (creates esp32pm.toml interactively), `esp32pm add [package]` (adds to toml without downloading). Config file is esp32pm.toml — parse with serde. Global cache at ~/.esp32pm/. Error messages must be actionable — not 'Error: 404' but 'Package bme280 not found. Did you mean bme280_idf?'"
How to use these prompts: At the START of every VS Code Copilot / Cursor / Claude session, paste the full context prompt. Then describe what you're working on that session. The AI will have all the architectural context it needs and won't suggest things that conflict with your decisions. Refresh the context every session — AI agents don't remember between sessions.
